核心问题：Agent 如何通过工具与链交互——从"看到链上状态"到"动手操作链上状态"。

---

## 章节要点

### 核心认知

Web3 Tool Use 是把 RPC、合约读取、交易生成、钱包确认、区块浏览器、DeFi 操作变成 Agent 可调用工具的过程。**真正难的不是"能调用"，而是权限、参数、模拟和日志。**

### 第一性原理

1. **模型可以选择工具，但工具必须用确定性边界限制模型** — 不要让 Agent 直接拼接任意 calldata 或调用任意地址
2. **读写分离** — 读链和发交易必须是不同工具、不同权限
3. **参数结构化** — chain id、contract address、method、args 不能埋在自然语言里
4. **日志不可省** — 每次工具调用都要记录输入、输出、时间、来源和错误

### 工具分类 & 风险等级

| 工具类型 | 难度 | 风险 | 关键要求 |
|---------|------|------|---------|
| RPC Tool | 初级 | 低（只读） | 返回需带 chain id、block number、RPC provider |
| Contract Read | 初级 | 低 | 注意读错网络/合约/ABI 不匹配 |
| Contract Write | **高级** | **高** | simulation + policy 检查 + 用户确认 |
| Wallet Tool | **高级** | **最高** | 连接/签名/发送/授权 必须分开 |
| Explorer Tool | 初级 | 低 | 提供可验证证据（tx hash、合约源码） |
| DeFi Tool | **高级** | **高** | 白名单 + 最大交易额 + 滑点 + 人工确认 |
| Tool Permission | 高级 | — | 分层：自动允许 → session key → 人工确认 |
| Tool Log | 中级 | — | 覆盖：输入、输出、时间、链 ID、确认人 |

### 写交易前 7 步检查链

chain id → 目标合约地址 → ABI method + args → value + 代币变化预估 → gas 估算 → simulation 结果 → policy 检查 + 用户授权

## 与 Web3 开发经验的对照

| Tool Use 概念 | 已有的 Web3 知识 | 新认知 |
|---|---|---|
| 工具读写分离 | 合约有 view 和 write 函数 | Agent 工具层要做硬分离，不是靠开发者自觉 |
| 写交易检查链 | 用 ethers/viem 发交易前有意识检查 | 需要制度化的 7 步流程，少一步都可能出问题 |
| 工具权限分层 | 多签/时间锁是链上权限管理 | Agent 层还需要"自动→session key→人工"的额外权限层 |
| 日志 | 链上事件是天然的审计日志 | Agent 工具调用日志是另一层审计——回答"模型看到了什么才做的决定" |

---

## 关键体会

上一章 Chain-aware Context 讲的是 **Agent 看到什么**（输入层），这一章讲的是 **Agent 能做什么**（执行层）。两章合在一起构成完整闭环：**见到的链上事实 → 工具化后的行动能力**。少任何一环，Agent 要么是瞎子，要么是没手。

另外这章真正在讲的是**安全工程**：Web3 工具的风险比普通 API 调用高一个量级，因为写交易会改变资产和权限。工具设计本质上是在**"模型能力"和"确定性边界"之间做权衡**。
