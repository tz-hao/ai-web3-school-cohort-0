## 一句话总结整个 Bridge

**AI × Web3 不是把两个 buzzword 拼在一起，是解决「AI Agent 要在链上做事」时必须回答的 15 个问题。** 每章一个问题，每个问题都是真正的交叉——纯 AI 和纯 Web3 都不回答。

## 15 个问题 × 15 章

| # | 章节 | 问题 | 一句话答案 |
|---|------|------|-----------|
| 1 | Chain-aware Context | Agent 怎么看到链上状态？ | 把链上事实（余额、合约、event）翻译成模型可读的上下文，带来源和时效 |
| 2 | Web3 Tool Use | Agent 怎么操作链上？ | 把 RPC、合约读、交易生成、钱包签名变成结构化工具，读写分离 |
| 3 | Agent Workflow | 哪些步骤自动，哪些人确认？ | 只读自动、草稿自动、签名和写链必须 human-in-the-loop |
| 4 | Machine Payment | Agent 怎么小额高频支付？ | Session Key 预授权 + Policy 合约链上强制限额，不靠 Agent 自觉 |
| 5 | Settlement & Escrow | 付款和交付有时间差怎么办？ | 资金锁在合约 → 交付 → 验证 → 释放/退款/仲裁，状态机兜底 |
| 6 | Agent Identity | 对面真的是这个 Agent 吗？ | 链上注册 + .well-known/agent-proof + DNS+TLS+签名三重验证 |
| 7 | Agent Trust & Reputation | Agent 的过往行为可查吗？ | 每次任务上链记录，成功/失败/返工/评分聚合为不可篡改的声音 |
| 8 | AI Oracle | 主观判断怎么变链上可用信号？ | AI 在 TEE 里做判定 → 结构化输出上链 → 24h 挑战窗口 + 仲裁兜底 |
| 9 | AI Security | Agent 被注入攻击怎么办？ | 三层防护：输入标记 + 上下文隔离 + prompt 级防御。Web3 注入能转走钱 |
| 10 | AI Privacy | Agent 的数据都去哪了？ | 7 项数据逐项追踪：什么存本地、什么上链、什么发给第三方 |
| 11 | AI Sovereignty | 用户还能控制自己的 Agent 吗？ | 数据可导出、模型可切换、权限可一键撤销、身份可跨平台验证 |
| 12 | Governance AI | AI 能帮我投票吗？ | 整理提案、对比论据、标注来源——可以。投什么票——不可以 |
| 13 | Decentralized AI | 推理一定要跑在中心化服务器吗？ | 多节点订单簿 + 共识验收 + TEE 证明 → 不信任任何单节点 |
| 14 | Dev Tooling | AI 怎么辅助合约开发？ | 读合约、提函数、列不变量、生成测试建议。不能说「安全」 |
| 15 | Open Track | 还有什么没被分类的问题？ | Agent 安全沙箱、钱包插件安全、跨链身份互认…留给你定义 |

## 三条贯穿全 Bridge 的主线

### 主线 1：权限分层

从 Web3 Tool Use 的「读写分离」到 Machine Payment 的 Session Key 限额，到 Escrow 的 Evaluator/Arbiter 分工，到 Agent Wallet 的「提案-确认」模式——**整个 Bridge 就是一层一层地细分「谁能做什么」**。

```
只读（自动）→ 草稿生成（自动）→ 小额支付（Session Key）→ 大额转账（人确认）→ 争议（仲裁）
```

### 主线 2：可验证性

从 Chain-aware Context 的 citation，到 Reputation 的链上 event，到 AI Oracle 的 TEE 证明，到 Decentralized AI 的多节点共识——**不能验证的 AI 输出在 Web3 里毫无价值**。每句结论都要能回到链上证据。

### 主线 3：人始终有最终控制权

从 Agent Workflow 的 human-in-the-loop，到 Escrow 的多签仲裁，到 Sovereignty 的一键 kill switch，到 Governance AI 的「AI 不替你做投票建议」——**自动化程度越高，撤回机制必须越强**。
