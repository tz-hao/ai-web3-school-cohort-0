## 一、记忆系统（Agent 的持久化大脑）

### 1. 三种记忆类型

Agent 的记忆不是一整坨，工程上分三层：

| 记忆层 | 一句话 | 生命周期 | Web3 类比 |
|--------|--------|----------|-----------|
| **短期记忆 (STM)** | 当前上下文窗口 | 聊完就忘 | 内存池（mempool）——只存当前这笔 |
| **工作记忆 (WM)** | 当前任务中间状态 | 任务结束释放 | 合约临时变量——函数执行完释放 |
| **长期记忆 (LTM)** | 持久化知识/偏好 | 跨会话存在 | 链上状态——持久化存储 |

> **核心原则**：短期解决"你在说什么"，工作解决"做到哪了"，长期解决"以前怎么说的"。

### 2. 长期记忆的存储方案

| 方案 | 怎么存 | 怎么取 | 适合 |
|------|--------|--------|------|
| 向量检索 (RAG) | 文本 → embedding → 向量库 | 语义搜索 | 模糊回忆、知识库 |
| 结构化存储 | 直接写 DB/JSON/文件 | 精确查询 | 事实性信息、配置 |

**向量检索流程**：文本 → embedding → [0.23, -0.87, ...] → 向量库 → top-K 近似匹配。

**切分策略（chunking）**：
- 按段落切（200-500 token/chunk）——最常用
- 按语义切（LLM 识别话题边界）
- 滑动窗口（重叠，避免切断关键信息）

> **经验法则**：需要"大概记得"用向量库，需要"精确知道"用结构化存储。不要把用户生日存成 embedding。

### 3. 两大核心挑战

**遗忘问题**：

| 情况 | 症状 | 解决 |
|------|------|------|
| 上下文溢出 | 聊着聊着"失忆" | 滚动摘要——长对话自动压缩 |
| 记忆膨胀 | 越堆越多、召回变慢 | 加权重/时间衰减/访问频率 |
| 记错东西 | 旧记忆回答新问题 | 记忆带时间戳，版本化 |

**冲突问题**：

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| 最新优先 | 时间戳排序，新盖旧 | 用户偏好型 |
| 来源加权 | 高权威来源 > 低权威 | 知识冲突型 |
| 显式确认 | 检测到冲突反问用户 | 高风险决策 |

> **Web3 类比**：冲突 ≈ 链分叉。最新优先 = 最长链原则，来源加权 = 验证者权重，显式确认 = 社会共识硬分叉。

> **经验法则**：偏好型用"最新优先"，事实型用"来源加权"，涉及钱的用"显式确认"。

## 二、Prompt 工程（让 Agent 从"靠运气"到"稳定可控"）

### 1. System Prompt vs User Prompt

| 层 | 放什么 | 更新频率 | Web3 类比 |
|----|--------|----------|-----------|
| System Prompt | 角色、规则、工具说明 | 几乎不变 | 合约代码——定义行为边界 |
| User Prompt | 具体任务、数据 | 每次不同 | 交易 calldata——每次不同输入 |

**核心原则**：把"你是谁""不能干什么"放 system，把"这次干什么"放 user。混着放 → Agent 随任务变来变去，"看心情"遵守规则。

### 2. 结构化输出

让 Agent 产出能被程序解析，不能靠"感觉"：

| 方式 | 做法 | 适用场景 |
|------|------|----------|
| Markdown 约定 | 约定格式，正则提取 | 简单场景，够用就行 |
| JSON Mode | 强制只输出 JSON | 程序消费，但会降低推理质量 |
| 函数调用 | 原生输出结构化参数 | 最可靠（部分模型支持） |
| 约束解码 | token 生成阶段强制语法 | 100% 符合，企业级 |

> **Web3 类比**：Markdown ≈ 注释写接口文档（看人品），JSON Mode ≈ ABI 编码，约束解码 ≈ solc 编译器（语法层保证）。

> **经验法则**：能用函数调用的用函数调用，不行用 JSON Mode，别用正则硬解析自然语言。
