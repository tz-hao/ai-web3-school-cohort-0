## 阅读章节

### Chain-aware Context — 链感知上下文

**核心认知：** Chain-aware Context 是让 AI Agent 在回答或行动前，能看见正确的链、地址、合约、交易、事件、余额、授权和数据来源，而不是只靠用户一句话猜测链上状态。

普通 AI 的上下文来自文档、聊天历史和网页。AI × Web3 多了一层：**链上状态持续变化，而且很多状态和资产、权限、交易执行直接相关**。如果 Agent 不知道当前 chain id、合约地址、ABI、用户授权、交易历史和数据更新时间，就可能给出错误建议，甚至生成危险交易。

链感知上下文的核心，是把链上事实变成模型可读、可引用、可验证的上下文。

---

### 第一性原理

1. **模型不能凭语言记忆判断链上事实** — 链上事实必须从工具和索引层读取。模型知道"Uniswap 是 DEX"没有用，真正执行时需要知道具体网络、具体合约、具体池子、当前价格、余额、allowance、滑点和模拟结果。

2. **链上状态有时间性** — 同一个地址的余额、授权和仓位会随区块变化。上下文必须携带数据读取的时间信息。

3. **上下文要带来源** — 合约地址、区块号、交易哈希、explorer 链接都应该能追溯。

4. **上下文要区分事实和解释** — 工具返回事实，模型负责解释，不要把模型猜测当事实。

---

### 关键知识节点

#### On-chain Data（初级）
链上可直接验证的数据：余额、交易、日志、合约状态、区块信息。来源包括 RPC、区块浏览器、索引器、协议 API。

> **Agent 读取时至少要带：** chain id、block number、contract address、method、返回值、读取时间。缺少这些字段，模型容易把不同链、不同时间和不同合约的数据混在一起。

#### Contract Docs（初级）
ABI 只能告诉你函数签名，不能解释业务语义。比如 `execute(bytes calldata data)` 可能是普通执行，也可能是高权限入口。文档、NatSpec、README、审计报告可以补足语义。

> **注意：** 文档可能过期。Agent 读取后仍需用链上数据验证：合约地址、版本、owner、proxy implementation、事件和最近交易是否匹配。

#### ABI / Event（中级）
ABI 让工具知道如何编码函数调用、解码返回值和解析事件。Event 是合约向外部系统留下的业务日志（Transfer、Swap、Deposit、VoteCast）。

> **安全提醒：** 能调用不等于应该调用。写交易前还需要权限、余额、allowance、slippage、simulation 和 policy 检查。

#### Transaction History（中级）
帮助 Agent 理解用户或合约过去做过什么：是否已授权、策略是否执行过、地址是否和高风险合约交互过。

> **证据要求：** 至少保留 tx hash、block number、from、to、method、value、token transfers、logs。模型可以总结，但证据必须可追溯回链上。

#### Explorer Context（初级）
区块浏览器提供的可视化链上证据——交易是否成功、合约是否验证、源码是否公开、事件是否发出。

> **在 AI 产品里，给 explorer link 比只给一句"交易成功"更可靠。**

#### Indexing Context（中级）
把链上事件整理成面向产品查询的数据。Agent 需要的不只是"某个区块有什么"，而是"用户最近 30 天的 DeFi 操作"这类查询。

> **索引上下文必须带时间戳和同步状态：** 落后 500 个区块的索引结果不应该被当成当前事实。

#### Citation（初级）
链上场景的引用可以是交易哈希、区块号、合约地址、event log、explorer 链接、文档 URL、审计报告章节。

> **没有 citation 的链上解释，只能算观点；带 citation 的解释，才有机会被验证和追责。**

---

### 在 AI × Web3 中的位置

Chain-aware Context 是所有链上 Agent 的**输入层**。没有这层，Web3 Tool Use、Agent Workflow、Agent Wallet 都会建立在不可靠上下文上。

一个好的链感知上下文包应该像这样：
- 用户目标
- 当前 chain id 和网络名称
- 用户地址和余额
- 相关合约地址、ABI、文档和风险提示
- 最近交易和授权
- 索引数据更新时间
- 每条关键结论的 citation

## 学习笔记

### 与 Web3 开发经验的对照

| Chain-aware Context 概念 | 我已有的 Web3 知识 | 新认知 |
|---|---|---|
| On-chain Data 需带时间戳 | 知道数据是链上公开的 | 没想过 Agent 把不同时间的数据混在一起的风险 |
| ABI 能调用≠应该调用 | 合约安全基本常识 | 这个检查应该放在 Agent 工具层，而不是靠模型自觉 |
| Citation 可追溯 | 做审计时习惯留 tx hash | 在 Agent 上下文里每条结论都要带来源是新的工程要求 |
| 索引有同步延迟 | 知道 The Graph 有延迟 | Agent 需要主动检查同步状态，不能假设索引是最新的 |

### 关键体会

这一章其实在讲一件事：**Agent 的"知情权"比人类的知情权要求更严格**。开发者扫一眼 Etherscan 页面就能直觉判断"这个合约有问题"，但模型没有这种直觉。它需要每一块事实都被明确标注——来源是什么、什么时候读的、是否已验证。少一个字段，模型就可能调用错误合约。

这和 Web3 开发里"显式优于隐式"的设计哲学是同一回事。
