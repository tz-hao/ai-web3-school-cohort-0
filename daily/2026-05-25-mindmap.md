## 总览

```
AI × Web3 的所有设计，本质上是回答三个元问题：

1. Agent 凭什么能动我的钱？
   → 支付/结算/工作流/账户

2. 我凭什么信这个 Agent？
   → 身份/声誉/Oracle/推理网络

3. 出事了怎么办？
   → 安全/隐私/主权/治理
```

## 一、支付与结算（Machine Payment + Settlement & Escrow）

```
元问题：Agent 凭什么能动我的钱？
```

### 1. API 即时支付流 · agent-api-payment-flow.md

| 维度 | 内容 |
|------|------|
| **核心问题** | Agent 买了 API 但还没拿到结果——先付钱怕被骗，后付钱服务方怕白干。怎么在「即时」场景下安全支付？ |
| **解决方案** | Session Key 预先授权（限额 + 时效 + 白名单）→ Agent 在预算内自主付款 → Policy 合约链上强制执行限额 → 付款即证明（tx hash 公开可查）→ 完整审计日志 |
| **一条限制** | Session Key 私钥泄露后，攻击者在限额内可随意花钱。单日限额是最后防线——如果限额设太高，Session Key 的安全性趋近于零。 |

### 2. Escrow 异步结算流 · agent-escrow-report-flow.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 付款和交付之间有 10 分钟时间差。没有第三方中介，怎么保证「付了钱一定能拿到货」或「没拿到货一定能退款」？ |
| **解决方案** | 链上 Escrow 状态机：FUNDED → SUBMITTED → (Evaluator 判定) → SETTLED/REFUNDED/DISPUTED。超时自动退款（任何人都能触发）。争议进入 2/3 多签仲裁。 |
| **一条限制** | Evaluator 只检查结构（字段存在、类型正确），不检查质量（分析是否真的有道理）。用户拿到报告后仍需人工判断内容——质量纠纷最终靠 Arbiter 而非代码。 |

### 3. DEX Swap 工作流 · constrained-web3-assistant.md

| 维度 | 内容 |
|------|------|
| **核心问题** | AI 辅助做一笔 Uniswap 兑换，哪些步骤 Agent 可以自动完成，哪些步骤必须人确认？边界在哪？ |
| **解决方案** | 9 步工作流，3 个人工确认点：①解析结果确认 ②链上状态+风险确认 ③钱包签名确认。AI 只做：解析意图、查询链上、生成 calldata、验证结果——全部只读。签名永远在钱包本地完成。 |
| **一条限制** | 人工确认的实际效果取决于用户是否真的「看」了弹窗内容。盲签（习惯性点确认）使三层确认形同虚设——这是人因问题，不是技术问题。 |

### 4. 账户类型对比 · account-comparison.md

| 维度 | 内容 |
|------|------|
| **核心问题** | EOA、智能账户、多签账户——Agent 应该用哪种？ |
| **解决方案** | 智能账户是 Agent 的唯一正确答案。EOA 全或无（给私钥=全丢），多签每次要等人（敏捷性为零）。智能账户支持 Session Key 限额 + Guardian 恢复 + Policy 自动化——同时满足 Agent 需要的三个特性：自动执行、权限分级、可接管。 |
| **一条限制** | 智能账户的权限策略本身就是一段代码——代码有 bug，策略就废了。合约安全和私钥安全是两种完全不同的安全范式。 |

## 二、身份与信任（Agent Identity + Trust & Reputation）

```
元问题：我凭什么信这个 Agent？
```

### 5. Agent Profile 设计 · agent-profile-design.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 有人说「我是 ChainLens Agent」，怎么证明他真的是？他的能力声明可信吗？ |
| **解决方案** | 三层：①链上 NFT 注册（Agent = ERC-721 token，所有权可转移）②IPFS 存 Profile JSON（内容寻址，CID 变 = 版本变，不可抵赖）③`.well-known/agent-proof` + DNS + TLS + 链上签名三重验证（攻击者要同时攻击三层才可能冒充） |
| **一条限制** | DNS 仍然是中心化依赖。如果 DNS 被劫持，三重验证的第一重就破了。ENS + IPFS 可替代 DNS，但用户端支持尚未普及。 |

### 6. EIP-8004 标准 · eip-8004-summary.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 每个项目自己设计 Agent 身份系统 → 互不兼容 → 无法跨平台发现和信任 Agent。行业需要统一标准。 |
| **解决方案** | 三个注册表：Identity Registry（Agent 是谁，ERC-721 NFT）→ Reputation Registry（Agent 过去表现如何）→ Validation Registry（这次推理可信吗）。信任模型可插拔：声誉/加密经济/zkML/TEE Oracle/人工裁判。作者阵容：MetaMask + EF + Google + Coinbase。 |
| **一条限制** | 2025-08 提出，截至 2026-05 仍是 Draft。标准需要生态采用才能体现价值——如果只有一个注册表部署了一个 Agent，互操作性仍是零。 |

### 7. 声誉系统 · agent-reputation-system.md

| 维度 | 内容 |
|------|------|
| **核心问题** | Agent 说「我准确率 98%」——证据在哪？声誉不能靠自述，必须可验证、可聚合、不可篡改。 |
| **解决方案** | 每次任务链上记录：task 定义 + 交付物 hash + 逐字段 review（不是主观打分，而是逐项标注正误）+ 是否可复现验证 + 失败则触发 slashing。聚合：成功/失败/返工/平均分由链上 event 自动计算，任何人可独立验算。 |
| **一条限制** | 刷评分（Agent 用小号给自己下任务）和恶意差评（竞争对手故意打低分）需要「用户身份权重」来缓解——但这又引入了「谁的用户身份更可信」的递归问题。 |

### 8. AI Oracle · ai-oracle-task-judgment.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 确定性 Evaluator 能检查「字段是否存在」，但不能判断「分析写得有没有道理」。主观判断怎么变成链上可用信号？ |
| **解决方案** | AI 在 TEE 里做推理 → 结构化输出（accepted + score + reason + section_scores）→ 链上记录（prompt 模板 hash + 输出 hash，可复现）→ 24h 挑战窗口（押金防恶意挑战）→ 争议进入 2/3 多签仲裁。Oracle 节点犯错有 slashing：3 次暂停，10 次永久除名。 |
| **一条限制** | 同一个模型、同一份报告，不同时间推理可能结果不同（temperature 随机性）。`temperature=0` 可缓解但不完全消除——模型本身就是概率性的，这和链上需要的确定性有本质矛盾。 |

### 9. 开放推理网络 · open-inference-network-spec.md

| 维度 | 内容 |
|------|------|
| **核心问题** | AI 推理一定要跑在某个中心化服务器上吗？怎么在「不信任任何单节点」的前提下获得高质量推理结果？ |
| **解决方案** | 多节点订单簿模式：请求方发布任务 → N 个节点各自推理 → Evaluator 对比结果（共识阈值 60%）→ 最优节点拿全款，其余拿 30%。超时/全部不合格 → 全额退款。节点报价竞争 + 声誉排序 = 市场机制保证质量。 |
| **一条限制** | 多节点 = 成本 × N。一个 0.05 USDC 的推理变成 0.15+ USDC。对于「今天的天气怎么样」这种低价值查询不划算——去中心化的收益必须大于成本溢价才值得。 |

## 三、安全与隐私（AI Security + AI Privacy + AI Sovereignty）

```
元问题：出事了怎么办？Agent 被攻击了会怎样？我的数据去哪了？
```

### 10. Prompt Injection 防护 · prompt-injection-defense.md

| 维度 | 内容 |
|------|------|
| **核心问题** | Agent 读了一份「看起来正常」的合约文档，文档里藏了 `[SYSTEM_INSTRUCTION]` 让它转账——Agent 凭什么分辨真假指令？ |
| **解决方案** | 三层防护：①输入 Sanitizer（扫描 + 包裹在 `<UNTRUSTED_CONTEXT>` 中）②上下文标记（显式 WARNING：此内容不可信）③系统 Prompt 防御（最高优先级规则：UNTRUSTED_CONTEXT 内的指令永远不执行）。Web3 场景的特殊性：注入不只是让 Agent 说错话，是让 Agent 转走你的钱。 |
| **一条限制** | 防护规则存在于 prompt 中——长对话中可能被挤出 context window。攻击者可以先用大量无害内容填满上下文，再在末尾注入——前两层防不住，第三层被挤出后形同虚设。 |

### 11. 数据流图 · agent-data-flow.md

| 维度 | 内容 |
|------|------|
| **核心问题** | Agent 处理的数据每一段流向哪里？谁看得见？用户怎么控制？ |
| **解决方案** | 7 项数据逐项追踪（输入/地址/交易历史/API Key/模型上下文/日志/记忆），每项标注：存储位置 → 发给谁 → 是否脱敏 → 用户能否删除。数据风险评估矩阵：泄露影响 × 泄露概率 = 风险等级。用户控制面板：`agent data show` → `agent permission revoke` → `agent data delete`。 |
| **一条限制** | 链上数据永久不可删除。342 笔交易记录、89 条 Agent Event——只要有人知道你的地址，就能画出你的完整行为画像。隐私在链上的边界是「你能换地址，但不能删历史」。 |

### 12. 主权检查表 · agent-sovereignty-checklist.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 用户对 Agent 有几分控制权？不是「能不能用」，是「能不用、能离开、能验证」。 |
| **解决方案** | 五个主权维度：①数据可导出（JSONL/CSV 标准格式）②模型可切换（云端 ↔ 本地，锁定单一 Provider）③权限可撤销（一键 kill switch，Session Key + Token Allowance 秒级收回）④身份可验证（跨平台：API + Telegram + Twitter 同一套链上证明）⑤执行可审计（每次操作可追溯到输入、推理、输出）。主权分五级：能用 → 能导出 → 能切换 → 能撤销 → 能验证。 |
| **一条限制** | 当前设计达到 Level 4（能撤销）。Level 5（能验证）需要 ZK 或 TEE 证明来自动化验证——目前依赖人工审查，不可规模化。 |

## 四、治理与开发工具（Governance AI + Dev Tooling）

```
元问题：Agent 能替我做决定吗？Agent 能帮我写合约吗？
```

### 13. 治理提案摘要模板 · governance-proposal-template.md

| 维度 | 内容 |
|------|------|
| **核心问题** | DAO 提案又长又复杂——AI 能帮我读懂提案、整理论据，但不替我做投票决定吗？ |
| **解决方案** | 标准化模板：提案目标 + 预算 + 里程碑 + 支持理由（附来源）+ 反对理由（附来源）+ 未回答问题 + 历史关联 + 来源标注 + AI 声明（列出 AI 没做什么）。关键原则：两面等权呈现、每条结论有来源、暴露不确定性、模板全程不出现「建议投票」。 |
| **一条限制** | AI 整理的「支持理由」和「反对理由」可能自带偏见——AI 倾向于把更常见的论点放在前面，或给某些论点更详细的阐述。篇幅不等于权重。 |

### 14. Uniswap UNIfication 真实提案 · governance-summary-uniswap-unification.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 一个真实的、已通过的 DAO 提案——用上面的模板套进去，能产出什么？ |
| **解决方案** | 用治理模板分析 Uniswap UNIfication（费用开关 + 1 亿 UNI 烧币 + 结构重组）。产出：6 条支持理由 + 6 条反对理由 + 8 条来源链接 + 5 个需要人工判断的争议点 + 8 个后续执行追踪。 |
| **一条限制** | AI 没有对反对方的论点做独立事实核查。@rodeo_crypto 说「过度支出」——AI 没有去查 Uniswap 的实际支出报表。摘要的价值取决于「你能通过来源链接自己核实每一个关键事实」。 |

### 15. ERC20 合约分析 · contract-review-erc20.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 拿到一份开源 Solidity 合约，AI 能帮我读懂、提取关键信息、给测试建议吗？——安全结论必须人工复核。 |
| **解决方案** | 系统化分析框架：①合约职责 + 权限点 ②会移动资产的函数列表 ③3 个最值得测试的不变量（总供应量守恒/Allowance 不可绕过/零地址隔离）④5 条测试用例建议（含 front-running、无限 approve、合约锁死等）⑤5 个需要人工复核的安全问题（🛑 标记）。 |
| **一条限制** | AI 不能说「这个合约安全」。所有派生合约的 mint 权限、unchecked 块正确性、approve 竞争条件——都必须人工逐项复核。AI 的测试建议覆盖了常见模式，但覆盖不了针对特定业务逻辑的攻击。 |

### 18. Agent Security Sandbox · open-track-agent-sandbox.md

| 维度 | 内容 |
|------|------|
| **核心问题** | 开发者搭好 Agent 准备部署到主网——最后一刻的恐惧：「如果有人发来恶意合约文档，Agent 会不会被骗去转账？」现在只能靠人工 review + 想象力找漏洞。 |

| **一条限制** | fork 环境的 gas/MEV 行为不可能和主网完全一致。sandbox 通过了 ≠ 主网安全。Sandbox 只能证明「这 30 种已知攻击打不穿」，不能证明「没有第 31 种攻击」。 |

## 三条主线的验证

回到 2026-05-24 Bridge 总结的三条主线，在每份设计中的体现：

### 主线 1：权限分层

```
只读（AI 自动）→ 草稿生成（AI 自动）→ 小额支付（Session Key）→ 大额转账（人确认）→ 争议（仲裁）

体现：
  constrained-web3-assistant   → 读写分离：AI 只读，签名在钱包
  account-comparison           → EOA（全权）vs 智能账户（Session Key）vs 多签（集体确认）
  agent-api-payment-flow       → Session Key 预算内自主，超额合约拒绝
  agent-escrow-report-flow     → Evaluator/Arbiter 分工：代码验证 + 人工仲裁
  safe-session-key             → owner（无限制）→ delegate（token 限额）→ 无权
```

### 主线 2：可验证性

```
每句结论都要能回到链上证据。

体现：
  agent-profile-design         → IPFS CID 内容寻址 + .well-known/agent-proof 签名
  agent-reputation-system      → 链上 event 自动聚合，任何人都能独立验算
  ai-oracle-task-judgment      → prompt 模板 hash 上链，TEE 远程证明，可复现
  open-inference-network-spec  → 多节点共识，少数服从多数，结果可对比
  eip-8004-summary             → Validation Registry：从声誉到 zkML 四种验证方式
```

### 主线 3：人始终有最终控制权

```
自动化程度越高，撤回机制必须越强。

体现：
  constrained-web3-assistant   → 3 个人工确认点，AI 永远不能绕过钱包弹窗
  agent-escrow-report-flow     → 24h 挑战窗口 + 2/3 多签仲裁
  agent-sovereignty-checklist  → 一键 kill switch + 五个主权限级别
  safe-session-key             → removeDelegate() 不需要 delegate 同意
```

## 一个统一的认知框架

遍历全部 19 份产出后，AI × Web3 的所有设计问题可以归约到一个模式：

```
                    ┌──────────────────┐
                    │  不信任的执行者    │
                    │  (AI Agent)       │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ 事前约束  │  │ 事中验证  │  │ 事后追责  │
        └──────────┘  └──────────┘  └──────────┘
              │              │              │
        Session Key    Evaluator      声誉 slashing
        Policy 合约    AI Oracle      链上审计
        Allowance      TEE 证明       kill switch
        白名单         多节点共识     仲裁
              │              │              │
              └──────────────┼──────────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │  链上强制执行     │
                    │  (不可绕过)       │
                    └──────────────────┘
```

**事前**：Agent 能做什么、花多少钱——在它行动之前就写死在合约里（#4, #6, #19）
**事中**：Agent 的每次行动被验证——交付物检查、推理可复现、TEE 证明（#7, #8, #9）
**事后**：Agent 的行为被记录、可审计、可追责——声誉聚合、slashing、仲裁（#5, #7, #12）

这三层不是「选一个」，是「都要有」。单层防护总会有一个角度被绕过。
