## 学习笔记

### 1. Chain-aware Context（链感知上下文）

**一句话**：让 AI 在推理前，能看见正确的链、合约、地址、余额、授权、交易历史和数据来源，而不是靠用户一句话猜。

**为什么 Web3 老手也需要关注这个**：你懂区块链，但 AI 模型不懂。模型只知道「Uniswap 是 DEX」，不知道当前在哪个链、哪个池子、你的余额、allowance、滑点。上下文层就是把这些链上事实翻译成模型能读的东西。

**六个知识节点**：

| 节点 | 难度 | 核心 |
|------|------|------|
| On-chain Data | 初级 | RPC/索引器返回的数据必须带 chain id、block number、contract、method、时间戳，否则模型会跨链混淆 |
| Contract Docs | 初级 | ABI 只告诉你函数签名，不告诉你语义。`execute(bytes)` 可能是普通执行也可能是高权限入口 |
| ABI / Event | 中级 | ABI 决定「能调什么」，Event 提供「发生过什么」。能调用 ≠ 应该调用 |
| Transaction History | 中级 | 用户/合约的过往行为，但证据必须能回到链上 tx hash |
| Explorer Context | 初级 | 给 explorer link 比 AI 自己说「交易成功」更可靠 |
| Indexing Context | 中级 | 链上事件按业务需求整理成查询。落后 500 个区块的索引结果 ≠ 当前事实 |
| Citation | 初级 | 每条关键结论必须有 tx hash / block number / explorer link 支撑，否则只是观点 |

**你已有的 Web3 直觉在这里的价值**：你天然知道 nonce、gas、block confirmation、event log 是什么，所以你在这里的重点不是学这些概念，而是理解 **Agent 拿到这些数据后怎么不犯错**。

### 2. Web3 Tool Use（Web3 工具调用）

**一句话**：把 RPC、合约读取、交易生成、钱包签名、区块浏览器变成 Agent 可调用的工具。真正难的不是「能调用」，而是权限、参数结构化、模拟和安全。

**设计原则（比具体实现更重要）**：

1. **读写分离**：读链上状态和发送交易必须是不同工具、不同权限。不可以有一个「万能 RPC 工具」同时读和写。
2. **参数结构化**：chain id、contract、method、args、value、slippage 不能埋在自然语言里让模型自由拼接。
3. **日志不可省**：每次工具调用必须记录输入、输出、时间、来源和错误。

**七个知识节点**：

| 节点 | 难度 | 核心 |
|------|------|------|
| RPC Tool | 初级 | 只读 RPC 可以开放更宽，写操作必须拆出去 |
| Contract Read | 初级 | view/pure 调用，最常用也最低风险。但 ABI 不匹配或数据滞后仍会误导 |
| Contract Write | 高级 | 必须经过 simulation → policy 检查 → 用户确认。不能给 Agent 「任意合约写入」能力 |
| Wallet Tool | 高级 | 最敏感边界。连接/签名/发送/授权/撤销必须分成不同动作 |
| Explorer Tool | 初级 | 提供可验证证据——交易成功？调了哪个方法？转出什么 token？ |
| DeFi Tool | 高级 | 需要协议白名单 + 最大交易额 + 滑点 + simulation + 人工确认 |
| Tool Permission | 高级 | 按工具/合约/方法/金额/时间/频率分层。大额转账必须人工确认 |
| Tool Log | 中级 | 每次调用记录：目标、工具名、输入、输出、错误、时间、chain id、tx hash、确认人 |

**为什么要这样设计？**

> 模型可以选择工具，但工具必须用确定性边界限制模型。不要让 Agent 直接拼接任意 calldata 或调用任意地址。

### 两章的关系

```
Chain-aware Context  ────→  Web3 Tool Use
    (输入层)                    (执行层)
    
Agent 先通过上下文层读到：
- 哪个链
- 哪个合约
- 用户余额/授权
- 最近交易历史
- 数据来源和时效

然后通过工具层执行：
- 只读查询（低风险）
- 交易草稿生成（中风险）
- 写链 + 签名（高风险，需确认）
```

**一句话总结今日**：上下文管「看到什么」，工具管「能做到什么」。不建好第一层，第二层的所有执行都会踩坑。

- 找一笔公开的交易哈希

- 标出哪些是链上事实，哪些是你的解释

**挑战**：设计一组 Agent Web3 工具
- 写一个只读工具（查 ETH 余额）
- 写一个合约读工具（查 ERC-20 allowance）
- 写一个交易草稿工具（生成 approve calldata 但不发送）
- 定义每个工具的输入 schema、输出字段、错误类型
