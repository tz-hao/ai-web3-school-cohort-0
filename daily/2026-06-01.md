## 二、深度知识：Few-shot & Chain-of-Thought

> 补 5/26 遗留的 🔜 项

### Few-shot Prompting

给 Agent 几个示例，让它学会"怎么做"而不是"做什么"。

| 类型 | 做法 | 适用 |
|------|------|------|
| Zero-shot | 不示例，直接问 | 简单任务 |
| One-shot | 给 1 个示例 | 格式固定 |
| Few-shot | 给 3-5 个示例 | 复杂推理/分类 |

**Web3 场景**：让 Agent 解释交易风险时，给 2-3 个范例（正常转账、approve 无限额、调用陌生合约），Agent 输出的风险等级就会更一致。

**坑**：示例太多 → 上下文溢出。示例有偏见 → Agent 学到错误模式。

### Chain-of-Thought (CoT)

让 Agent "把思考过程写出来"，不是直接给答案。

```
❌ "这个提案该不该通过？" → "通过"
✅ "这个提案该不该通过？一步一步分析：
     1. 提案金额是否在预算内？
     2. 收款地址是否在白名单？
     3. 调用的合约是否已知？
     4. 综合判断 → 通过/拒绝/需人工确认"
```

**变体**：
- **Zero-shot CoT**：加一句 "Let's think step by step" 就触发推理链
- **Auto-CoT**：自动生成示例 → 自动构建 CoT
- **Tree-of-Thought**：多分支推理 → 选最优路径（成本高，谨慎用）

**Web3 价值**：链上操作不可逆，CoT 让 Agent 在"动手"前先过一遍安全检查清单。

## 三、深度知识：Agent 错误处理

### 错误分类三层

| 层 | 例子 | 策略 |
|----|------|------|
| **可恢复** | RPC 超时、API 限流 | 重试 + 退避 |
| **需降级** | 工具不可用、模型切换 | 换工具/模型 |
| **必须停** | 超预算、未知合约、权限不足 | fail closed + 通知人 |

### 重试策略

```
简单重试：失败 → 等一下 → 再试（最多 N 次）
指数退避：失败 → 等 1s → 等 2s → 等 4s → 等 8s → 放弃
断路器：连续失败 M 次 → 暂停一段时间 → 再试一次
```

**Web3 特殊规则**：
- 交易已广播但 pending → **不要重发**（会双花），等确认或 replace with higher gas
- RPC 返回 error → 先验证 nonce 是否正确，再重试
- 合约 revert → 不要重试（同样的输入同样的结果），先查原因

### Fail Closed vs Fail Open

```
Fail Open  = 出错了继续（风险大，不适合 Web3）
Fail Closed = 出错了停止（默认策略）

SafePay 原则：所有链上操作默认 Fail Closed。
Agent 不确定的时候 → 停下来 → 问人。
```

### 错误信息设计

Agent 报错要包含：
1. 发生了什么（不是 "Error" 而是 "RPC 超时，第 3 次重试"）
2. 影响范围（"只影响本次查询，不影响已广播的交易"）
3. 下一步（"已暂停，等待你确认是否继续"）
