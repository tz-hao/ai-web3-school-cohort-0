### 1. LLM — 大语言模型

**核心认知：** LLM 是概率模型，不是真相源。它生成概率上合理的输出，不是天然可信的事实。

**关键概念：**
- **Token** — 模型处理文本的基本单位。代码、JSON、混合语言文本比普通段落更吃 token。上下文能放多少、成本多少、模型能不能完整看见关键信息，都由 token 决定。
- **Embedding** — 把文本映射成向量的技术，用于衡量语义接近度。适合搜索、聚类、推荐和 RAG，但不适合单独判断结论是否正确。
- **Transformer** — 核心架构，依赖 attention 机制。擅长在上下文中找模式，但不等于拥有稳定数据库。
- **Hallucination（幻觉）** — 模型生成合理但不真实的内容。在带执行能力的系统里会变成流程风险。处理方式：来源引用 + schema 校验 + 规则检查 + 人工确认。
- **Multimodal（多模态）** — 可处理文本、图片、音频等。价值在于读懂真实工作界面（图表、控制台、错误截图），但关键判断仍要回到结构化数据。

**Web3 视角的注意点：** LLM 是理解和生成层，负责把用户目标转成可讨论的计划、把链上数据解释成人能读的语言。真正的产品需要配合数据层（RPC、索引器）、编排层（Prompt、Context、Agent workflow）、执行层（工具调用、钱包、Smart Account）和安全层。

**最小实践：** 交易解释器 —— 输入交易哈希，让 LLM 读取交易详情和事件日志，输出解释。重点是把模型生成、链上事实、来源边界、不确定性分开。

### 2. Prompt — 提示词

**核心认知：** Prompt 是模型接口设计。它把任务目标、输入边界、输出格式、失败处理和安全规则写进可执行的沟通协议。**Prompt 是软约束，不是安全边界** —— 真正的边界必须由代码、权限、校验和审计承担。

**关键概念：**
- **Instruction（指令）** — 告诉模型角色、任务、不能做什么、输出格式。在真实产品里要区分"解释"和"执行"。
- **Few-shot（少样本）** — 放少量示例让模型模仿判断方式和格式。适合"风格和边界难一句话说清"的任务，但需要维护成本——示例要跟 eval 一起维护。
- **Structured Output（结构化输出）** — 让模型返回固定结构（JSON、函数参数）。不是为了好看，而是为了让后续代码能检查、拒绝、记录和回归测试。
- **Prompt Injection（提示注入）** — 攻击者通过用户输入、文档、检索内容让模型忽略原始规则。应对方式：外部内容标记为不可信 + 工具调用前参数校验 + 敏感操作走 allowlist + human approval。

**Web3 视角的注意点：** Prompt 不应该独自承担安全。更稳的链路是：Prompt 定义任务 → Context 提供可信数据 → 模型生成 → Code 校验 schema → Guard/simulation 检查链上影响 → Human check 确认高风险动作。

**最小实践：** 交易风险摘要 prompt。输出固定 JSON（summary, asset_changes, risk_level, requires_human_approval, uncertainties）。测试普通转账、无限授权、意图不匹配三种场景。

### 3. Context — 上下文

**核心认知：** Context 是模型这次能看到、能用、能被影响的信息空间。真正难的不是塞更多内容，而是把系统规则、用户目标、历史状态、工具结果和外部文档分清楚。

**关键概念：**
- **Context Window（上下文窗口）** — 模型一次请求能处理的最大范围。窗口大不意味着模型会完美使用每个细节——长上下文常见问题是"看见了但没抓住重点"。
- **Context Engineering（上下文设计）** — 设计上下文进入模型的方式。稳定工具型 Agent 的上下文应包括：任务状态、工具结果、证据、可信来源、外部检查、用户意图、系统禁止事项。
- **Memory（记忆）** — 跨请求保留的信息（偏好、历史任务）。容易带来隐性风险——用户过去允许某个动作，不代表现在仍然允许。不能替代实时授权。
- **Knowledge Base（知识库）** — 系统可检索的外部知识库。需要维护来源、更新时间、版本和废弃状态。

**Web3 视角的注意点：** 靠谱 Agent 上下文分层：指令层 → 任务层 → 事实层（链上状态、工具结果） → 知识层 → 记忆层。层级越清楚，越容易做权限控制、prompt injection 防护和审计。

**最小实践：** 钱包授权检查 Agent 的 context spec —— 明确模型回答前必须获取的链上状态（chain id、token 合约、allowance、spender 可信列表、simulation 结果），并区分实时查询字段 vs 缓存字段。

---

### 4. RAG — 检索增强生成

**核心认知：** RAG 让模型在生成回答时查询外部知识源，解决知识过期和"我不知道"的问题。核心链路：用户查询 → 检索 → 检索结果注入上下文 → 模型生成。

**关键实践：**
- Chunking（分块策略）和 Embedding 的选择直接影响检索质量。
- 检索结果需要标注来源和可信度——不能把论坛帖子和官方文档放在同一优先级。
- RAG 不能自动保证正确——它只是把外部知识更稳定地送进模型。
- 在 Web3 场景里，RAG 适合把合约源码、EIP 标准、审计报告、治理提案等文档纳入检索范围。

**闭环：** 知识库 → 向量化 → 检索 → 注入 prompt → 模型输出 → 来源校验。

---

### 5. Agent — 智能体

**核心认知：** Agent 让模型从"回答问题"进入"执行流程"。核心是模型 + 工具 + 循环（推理 → 行动 → 观察）。

**关键概念：**
- **Tool Use（工具调用）** — 模型决定调用哪些函数、传什么参数，而不是只输出文本。工具返回值进入下一轮推理。
- **ReAct（推理-行动循环）** — 模型不断进行"思考 → 行动 → 观察结果 → 再思考"的循环。
- **Human-in-the-loop（人在回路）** — 涉及资产变动、权限变化或不可逆操作的步骤，必须加入人类确认节点。
- **有限授权原则** — Agent 拿到的最小工具集和最小权限，而不是全部可用工具。

**Web3 视角的注意点：** Agent 在 Web3 场景里要处理的状态变更不可逆——一笔错误转账不能"撤回"。所以执行前模拟（simulation）、执行中分步确认、执行后审计日志，是 Agent 系统设计的三道防线。

---

### 6. MCP — 模型上下文协议

**核心认知：** MCP（Model Context Protocol）是模型与外部工具/数据源之间的标准化协议，让模型发现可用工具、调用工具、获取结构化上下文。

**关键要点：**
- 协议层解决的问题：每个 Agent 框架各自实现一套工具接入方式，MCP 提供统一标准。
- Server 提供工具和数据源，Client（模型/AI 应用）通过协议发现和调用。
- 与 Web3 的关系：MCP server 可以封装 RPC 调用、合约交互、钱包签名等能力，为 Agent 提供标准化的链上接口。
- 安全方面：MCP 需要结合权限控制，不能让模型随意调用所有暴露的工具。

---

### 7. Frameworks — 框架

**核心认知：** 框架解决的是编排层问题——把 LLM 调用、工具管理、上下文组装、状态保持和控制流组织成可维护的系统。

**常见框架定位：**
- LangChain — 最早的 LLM 应用框架，丰富的集成生态。适合快速原型，但抽象层多，调试成本高。
- LangGraph — 在 LangChain 基础上提供有向图执行，适合复杂多步 Agent 工作流。
- OpenAI Agents SDK — 轻量级官方 Agent 框架，适合直接在 OpenAI 生态里构建。
- 其他（Vercel AI SDK、DSPy 等）— 各有所长，选择标准看生态匹配度和控制力需求。

**选择原则：** 框架帮你加速原型，但也带来抽象依赖。从最小可行编排开始，只在需要时引入框架。

---

### 8. Vibe Coding — 氛围编程

**核心认知：** Vibe Coding 指的是利用 AI 辅助做快速原型开发——写 prompt 让模型生成代码，靠试错和迭代推进。

**关键要点：**
- 适合场景：快速验证想法、Hackathon 原型、UI 界面搭建、脚本工具。
- 不适合场景：生产级合约开发、安全敏感代码、复杂系统架构。
- 在 AI × Web3 场景里的价值：快速搭建前端交互界面、数据看板、演示 Demo，但链上逻辑（合约、权限、签名）仍需人工审查。
- 核心理念：让 AI 帮你加速"能做出来"的部分，自己专注在"不能错"的部分。

---

### 9. Evaluation — 评估

**核心认知：** 评估（Eval）是让模型输出、Agent 行为和工具调用可以被测试、回放和持续改进的体系。

**关键要点：**
- 没有 eval 的 AI 应用等于没有测试的软件——你不知道它什么时候会坏。
- Eval 维度：准确性、格式符合度、安全性、回退行为、延迟和成本。
- 与 Web3 的结合：交易解释是否正确识别了风险？Agent 是否在需要时触发了 human-in-the-loop？Prompt injection 检测是否灵敏？
- 评估集需要持续维护——新合约、新标准、新攻击模式出现后，旧的测试用例可能不再有效。

---

### 10. Fine-tuning — 微调

**核心认知：** Fine-tuning 是在基础模型上用特定数据集做额外训练，让模型在特定任务上表现更好。

**关键要点：**
- Fine-tuning 适合：固定输出格式、特定领域术语理解、风格固化。
- Fine-tuning 不适合：让模型拥有训练数据之外的知识（这个靠 RAG）、修复 prompt injection（安全层面不能靠训练解决）。
- 对大部分 AI × Web3 产品来说，RAG + prompt engineering 优先于 fine-tuning。只有当你能收集大量高质量的特定任务数据（例如交易注释对、合约审计报告分析）时，fine-tuning 才有性价比。
- 资源门槛高，数据质量比数据量更重要。

---

### 11. Inference — 推理服务

**核心认知：** Inference（推理服务）是模型部署和运行的过程——把训练好的模型以 API 或本地服务形式提供给应用使用。

**关键要点：**
- 推理方式：云端 API（OpenAI、Claude 等）、本地推理（ollama、vLLM、llama.cpp）、去中心化推理。
- 对 Web3 场景的特殊价值：去中心化推理（如 Phala、Bittensor 等）让推理过程可验证、抗审查，适用于需要可信执行环境的场景。
- 关键考虑：延迟（链上交互需要快速响应）、成本（高频调用费用）、隐私（数据是否经过第三方）。
- **去中心化 AI（Decentralized AI）**作为进阶方向：把推理、训练或验证从单点服务移向去中心化网络。

## 学习笔记

### 跨章节的核心线索

作为 Web3 开发者进入 AI 领域，以下是我理解的贯穿线：

1. **模型不能替代校验** — 从 LLM 的幻觉、Prompt 的软约束、Context 的可信分层到 Evaluation 的回归测试，所有章节反复强调同一个原则：模型输出是候选结果，不是事实本身。

2. **分层架构是安全感来源** — Agent 系统需要明确分层：数据层（链上状态）→ 编排层（Prompt + Context + RAG）→ 推理层（LLM）→ 执行层（工具 + 合约）→ 安全层（Guard + simulation + human check）。每一层的职责和边界越清楚，系统越安全。

3. **结构化胜过自由文本** — 从 Structured Output、函数调用到 Eval 的 schema，只要涉及后续自动化处理，都应该输出结构化数据，而不是散文。

4. **"最小实践"思维** — 每个章节都配了一个最小实践练习（交易解释器、交易风险摘要、wallet context spec 等），这些练习不是花哨的 demo，而是每个概念在 Web3 场景里最直接的落脚点。

### 和 Web3 的交叉点

| AI 概念 | Web3 对应 / 交叉点 |
|---------|-------------------|
| Hallucination（幻觉） | 链上交易不可逆，幻觉=资金风险 |
| Prompt Injection | 恶意 dApp 页面作为不可信输入 |
| Context Engineering | 链上状态 / blobs / logs 作为可信事实层 |
| Agent Tool Use | RPC、合约调用、钱包签名作为工具 |
| Agent Workflow | 交易模拟 → 签名 → 确认 → 结算的流程化 |
| Human-in-the-loop | 每一笔交易、每一次 approve 的确认节点 |
| MCP | 标准化 Web3 工具接口（RPC、签名、查询） |
| Evaluation | 交易解释正确性、Agent 行为安全性的回归测试 |
| Verifiable AI / Inference | 去中心化推理作为可信执行环境 |
