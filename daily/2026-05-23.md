## Machine Payment（机器支付）

### 一句话

机器之间如何完成小额、高频、无人值守的支付——不需要人每笔点确认。

### 核心理解

**普通支付**：人 → 打开钱包 → 填地址 → 输金额 → 签名 → 等确认。每笔都要人。

**机器支付**：
- 机器 A（Agent）需要调机器 B（API 服务）的接口
- 一次调用 0.005 USDC，一天可能调几百次
- 人不可能守在旁边每笔确认
- 必须有自动化支付通道

### 两种模式

| 模式 | 做法 | 适用 |
|------|------|------|
| Session Key | 智能账户签一个限额 Session Key，Agent 在额度内自由支付 | Agent 调用付费 API、高频查询 |
| 支付通道 | 双方在链上开通道，链下记账，最终结算才上链 | 两台服务器之间的持续计费 |

### 和传统支付的三个关键区别

1. **不需要每笔签名**：Session Key 预授权代替逐笔确认
2. **金额极小**：可以是 $0.001 级别——传统支付的手续费都不止这个数
3. **可以程序化触发**：Agent 在代码里判断「需要调这个 API → 查余额够 → 自动付 → 拿结果」

### 安全边界

> 机器支付的安全不靠 Agent 自觉，靠链上 Policy 合约强制执行：单笔上限、日累计上限、白名单收款方、Session Key 过期时间。

## Settlement & Escrow（结算与托管）

### 一句话

当付款和交付之间有**时间差**时，需要托管合约保证「一手交钱、一手交货」。

### 核心理解

**Machine Payment 的前提**：付款后立刻拿到结果（API 调用、查询）。服务方在线，交付是即时的。

**但当交付不是即时的**：
- 找人生成一万字的调研报告 → 需要 30 分钟
- 找设计师做 logo → 需要 2 天
- 找审计公司审合约 → 需要 2 周

「先付钱再等交付」→ 对方可能跑路。「先交付再付款」→ 对方不敢先干。

**Escrow 解决这个**：钱锁在合约里，双方都动不了。交付完成 → 自动释放。超时 → 自动退款。有争议 → 仲裁。

### 状态机（核心设计模式）

```
IDLE → FUNDED → SUBMITTED → SETTLED ✅
                  ↓              ↑
               timeout()    arbitrate(✅)
                  ↓              ↑
              REFUNDED ← DISPUTED
```

每个状态只能按箭头方向转移，不能回退。

### 三个关键角色

| 角色 | 做什么 | 谁来当 |
|------|--------|--------|
| **Evaluator** | 检查交付物是否符合要求 | 可以是确定性合约（检查 JSON 字段），也可以是 AI Agent（检查报告质量） |
| **Arbiter** | 争议时裁决 | 多签（3 人中 2 人同意），防止单点贿赂 |
| **Escrow 合约** | 资金托管 + 自动结算 | 链上合约，任何人可调用 timeout() |

### 两个反常识点

1. **任何人都能调 `timeout()`**——不是 Bug，是 Feature。Deadline 是公开事实，超时就该退款。不让 Payer/Payee 单方面控制退款时机。
2. **Evaluator 不检查质量，只检查结构**——「报告有没有包含 protocol_name 字段」vs 「报告写得好不好」。前者可自动化，后者靠人确认 + 争议仲裁兜底。

## 两章的连接

```
Machine Payment          Settlement & Escrow
─────────────────────────────────────────────────
同步交付（立刻拿到结果）   异步交付（等结果）
Session Key 授权          链上托管合约
支付=交付                  支付≠交付（有时间差）
每笔有 tx hash            每笔有 event log + IPFS hash
适合 API 调用             适合任务外包
不需要仲裁                需要仲裁兜底

        两者共同组成 Agent 的支付基础设施
```
