## AI 基础概念整理

### 1. LLM（Large Language Model）

LLM 本质是一个**基于海量文本训练的高维词汇概率系统**。它不是"会思考的大脑"，而是一个极度擅长"根据上文猜下一个词"的预测引擎。GPT、Claude、DeepSeek 这些模型，底层都是同一个原理：给定前面的词，预测下一个最可能的词。反复滚动，就生成了有逻辑的文本。

**反常识点**：
- LLM 并不"理解"你说的内容——它只计算概率分布。
- 同一个 prompt 多次运行，输出可能不同（因为采样温度 `temperature` 有随机性）。
- LLM 没有"真实世界知识"——它只知道训练数据中出现过的统计模式，所以会编造事实（幻觉）。

**实际例子**：
用 `deepseek-chat` 写 shell 脚本时，它生成过一个 `find . -delete` 命令，这种写法如果用在生产环境会直接删库。LLM 不懂"这个命令危险"，它只是觉得 `find ... -delete` 在训练数据中经常出现在一起。所以需要人审再执行。

**复核方式**：把 LLM 生成的代码/文本，再用另一条消息问"这段代码有什么安全隐患？"做交叉验证。

---

### 2. Prompt（提示词）

Prompt 不是"问问题"，而是**给 LLM 设置一个上下文约束空间**。Prompt 的质量决定了模型在"下一个词预测"这个无限可能中，被收敛到哪个方向。

一个优秀的 prompt = 角色 + 上下文 + 任务 + 格式约束 + 示例。缺了任何一块，模型就会用训练数据中的默认模式来填补——而那个默认模式大概率不是你想要的。

**常见误区**：
- ❌ 越短越好 → 足够的上下文比简洁更重要
- ❌ 一次性搞定 → prompt 经常需要迭代 3-5 轮才能稳定
- ❌ 加 "请严格遵循" 就有用 → 示例（few-shot）比指令管用得多

**对比举例**：

弱 prompt：
```
写一个 Python 函数反转字符串。
```
可能输出 `return s[::-1]` 但没有类型提示、docstring、错误处理。

强 prompt：
```
写一个 Python 函数反转字符串。要求：
1. 使用函数签名 def reverse_str(s: str) -> str
2. 包含完整的 docstring（参数，返回值，例子）
3. 处理空字符串和 Unicode 代理对
4. 不使用 s[::-1]（手动循环实现）
```
后者输出的代码直接可用。

---

### 3. Context Window（上下文窗口）

Context window 是 LLM **一次能"看到"的 token 总量**。它不是缓存，不是内存——每一次对话，模型都把整个上下文从头读一遍，然后生成下一个 token。

**关键认知**：
- 每一条消息 + 每一次回复，都在消耗这个窗口。
- 窗口满了，**最早的内容会被截断**（或丢弃），模型就"忘记"了对话开头的内容。
- 这就是为什么长对话末尾的模型表现会明显下降——指令可能已经在窗口外面了。

**管理技巧**：
- 关键指令放在对话**末尾**而不是开头（最新内容不会被裁）
- 长文档用参考方式而不是直接粘贴（MCP 文件读取 vs 全文粘贴）
- 定期 reset 并在新对话中植入关键 context（persistent memory 的作用就在这里）

---

### 4. Agent（智能代理）

Agent 不是"更聪明的 LLM"。Agent = **LLM + 工具 + 循环** 的复合体。

| | LLM | Agent |
|---|---|---|
| 做了什么 | 生成文本 | 观察 → 思考 → 行动 → 观察循环 |
| 能做什么 | 回答问题 | 执行命令、调用工具、自主决策 |
| 有记忆吗 | 只有 context window | 可以有持久化记忆 |
| 有工具吗 | 无 | 文件系统、代码执行、网络搜索等 |

**实际工作流（以 Hermes Agent 为例）**：
1. 用户说："检查 GitHub 上有没有新 issue"
2. Agent 的 LLM 分析这个请求 → 决定调用 `terminal` 工具运行 `gh issue list`
3. 看到输出 → 决定可能还需要 `search_files` 查看本地配置
4. 综合所有信息 → 生成最终回答

**常见误解**：
- ❌ Agent 就是 LLM 加了个壳 → Agent 的精髓是**自主决策循环**和**工具编排**
- ❌ Agent 能完全自主运行 → 需要清晰的边界和约束
- ❌ Agent 越"聪明"越好 → 一个有误导性的 Agent 比笨但安全的更危险

---

### 5. Tool Use（工具调用）

Tool use 是 **LLM 与外部系统交互的桥梁**。单独的 LLM 是"关在一个黑屋里的一台打印机"——它能生成很好的文本，但对外界一无所知。Tool use 给它开了几扇窗户。

**Hermes Agent 的工具模型**：
```
用户输入 → LLM 分析 → 选择工具 → 调用工具 → 
工具结果回到 LLM → LLM 生成最终回复
```

这个循环在模型看来是这样的：
1. 模型看到你的指令
2. 模型生成一个特殊的格式（如 JSON 格式的 function call）
3. 运行时解析这个格式，执行对应的工具
4. 结果以系统消息的形式"喂"回给 LLM
5. LLM 基于工具输出继续生成回答

**关键认知**：**LLM 本身并不能真正执行任何操作**。它只是说"我建议调用这个工具，参数是这些"。执行权在 Agent 运行时中。这意味着：
- **安全边界**：如果 prompt 注入，模型可能建议危险的命令。Agent 运行时需要自己的安全策略。
- **可靠性**：模型可能会编造工具的名字（幻觉），好的 Agent 需要函数定义验证。
- **延迟**：每次工具调用就是一个完整的 LLM 回合，多个工具调用会显著增加响应时间。

---

### 6. AI Coding

AI Coding 不是我让 AI 写代码那么简单。它是一种**新型人机协作范式**。

**模式分类**：

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| 补全（Completion） | 写一行提示，AI 补几行 | IDE 插件（Copilot、Codeium） |
| 对话（Chat） | 描述需求，AI 生成完整函数/文件 | 明确但重复的任务 |
| Agent 模式 | AI 自主：读代码→改代码→运行→修复 | 复杂任务、多文件改动 |
| 审查（Review） | AI 对已有代码提意见 | PR Review、安全审查 |

**误区 1：AI 写的代码可以直接用** → 事实：AI 代码正确率大概在 70-80%。剩余的包括：API 版本不对、类型错误、忘了 import、逻辑边界有 bug、安全隐患。必须修改和测试。

**误区 2：AI 适合一切语言和框架** → 事实：AI 在主流语言和框架（Python、React、TypeScript、Django）上表现最好。小众或冷门框架的表现会断崖式下降。

**误区 3：prompt 越详细越好** → 事实：太长且结构混乱的 prompt 反而会稀释关键信息。结构化 > 详细化。用 bullet point 分条，每条不超过一句话。

**最佳实践 prompt 模板**：
```
# 角色
你是一个资深 Python 后端工程师

# 任务
重构这个函数，让它使用 async/await

# 约束
- 保持相同的函数签名
- 保留所有 docstring
- 只修改实现逻辑

# 上下文
文件：/path/to/handler.py
```

## Web3 基础概念整理

> 用自己的话重新表述 9 个核心概念，为测试网交易、合约调用、钱包确认和链上验证任务做准备。

### 1. Account（账户）

**一句话**：链上所有操作的发起者和接收者，分 EOA（人控）和合约账户（代码控）两种。

**例子**：你的 MetaMask 里以 `0x` 开头的身份就是 EOA——用它签名、发交易、持有资产。Uniswap V2 Router 的地址 `0x7a250d5630B4cF539739dF2C5dAcb4c659F2488D` 是合约账户——你调它的 `swapExactTokensForETH`，它按代码逻辑执行。

**误区**：EOA 只能由私钥控制、合约只能由代码控制。Account Abstraction（ERC-4337）正在模糊这条边界——智能账户可以同时具备两者特性。Bridge 章节会反复用到。

---

### 2. Address（地址）

**一句话**：链上实体的唯一标识符，20 字节，通常表示为 `0x` + 40 个 hex 字符。

**例子**：Uniswap V2 Router = `0x7a250d5630B4cF539739dF2C5dAcb4c659F2488D`。把 ETH 发到这个地址 vs Vitalik 的地址，链上行为完全不同。地址没有内置校验和，EIP-55 混合大小写只能防 typo，挡不了剪贴板劫持。

**安全提醒**：Agent 生成交易时，合约地址必须来自链上查询或已验证 explorer，绝不能靠模型「记得」。始终用区块浏览器交叉验证。

---

### 3. Wallet（钱包）

**一句话**：钱包不是「装币」的地方，是私钥管理器 + 签名器 + 链上交互入口。

**例子**：MetaMask 弹出确认窗口时，它在用私钥对交易做本地签名。币不在 MetaMask 里——它们一直在链上。钱包只是帮你生成签名证明你有权移动它们。

**安全提醒**：钱包确认弹窗是最后防线。Agent 生成的交易，点确认前必须看清 `to`、`value`、`data` 编码了什么。**永远不要盲签**——这就是 Web3 Tool Use 章节强调「高风险动作必须回到用户确认」的原因。

---

### 4. Private Key（私钥）

**一句话**：控制一个地址的终极凭证，32 字节随机数，谁有私钥谁就拥有该地址的全部资产。

**例子**：MetaMask 助记词派生出一组私钥。`eth_sign` 签名时私钥在本地运算，不发送到网上。但如果你把私钥粘贴到钓鱼网站，对方即刻拥有你的地址。

**安全提醒**：
- 私钥 ≠ 助记词。助记词是私钥的人类可读备份（12/24 个单词）
- **绝对不要在 public repo、聊天记录、prompt 里放私钥或助记词**
- Agent 管理私钥必须走 Smart Account / Session Key / MPC，不能明文持有

---

### 5. Transaction（交易）

**一句话**：改变链上状态的唯一方式，包含 `from`、`to`、`value`、`data`、`gas`、`nonce`、签名七个要素。

**例子**：Uniswap 上换 100 USDC → ETH，链上实际发生两笔交易：先 `approve` Router 使用你的 USDC，再 `swapExactTokensForETH`。每笔都不可逆。

**误区**：交易 ≠ 转账。`value = 0` 但 `data` 里编码了 `approve(spender, unlimited)` 的交易——对方可以随时转走你所有 token。Agent 工具必须区分：只读查询、交易草稿、写链发送——三种动作风险完全不同。

---

### 6. Gas（燃料费）

**一句话**：执行交易的计算成本，`gas limit × gas price = 实际费用`，支付给网络验证者。

**例子**：简单 ETH 转账消耗 21,000 gas。一笔 Uniswap swap 消耗 150,000–300,000 gas。EIP-1559 后 gas 分 base fee（燃烧）和 priority fee（小费）。gas price 高时同一笔操作可能从 $2 飙到 $20。

**误区**：交易失败 gas 照样扣——节点执行了你的代码直到 revert 点。Agent 生成交易前必须先做 gas 估算和 simulation。这也是 Web3 Tool Use 把 gas 估算列为写交易前置条件的原因。

---

### 7. Smart Contract（智能合约）

**一句话**：部署在链上的可执行代码，确定性地执行——相同输入永远相同输出。

**例子**：ERC-20 USDC 合约的 `transfer`、`approve`、`transferFrom`。任何人调 `transfer`，合约检查余额→扣款→增款→发事件，全程自动，不需要银行审批。

**误区**：合约代码「公开已验证」≠ 你读得懂、≠ 逻辑安全。`verified` 只说明源码和 bytecode 匹配。Agent 读合约时，ABI 告诉你「能调什么」，不告诉你「该不该调」——需要文档补足语义 + 链上数据交叉验证。

---

### 8. Testnet（测试网）

**一句话**：和主网相同协议但币没价值的平行网络，用于开发测试、零成本试错。

**例子**：以太坊 Sepolia 测试网——从水龙头领免费 Sepolia ETH，部署合约、发交易、调方法，全程零成本。出错了重来。

**误区**：测试网通过 ≠ 主网 OK。测试网的 gas、拥堵、MEV、预言机价格和主网完全不同。**测试网代币不要当真币交易**——有人在 Discord 骗新人买所谓「会上主网的测试币」。

---

### 9. Block Explorer（区块浏览器）

**一句话**：区块链的搜索引擎——查交易、合约、地址、区块、事件、代币。

**例子**：Etherscan 上输入你的地址，看到每笔交易、余额变化、持有的 token、交互过的合约。输入合约地址，看到源码（如 verified）、事件日志。

**误区**：Explorer 能显示假 USDC——Token 名可以叫「USDC」但合约地址不同。**验证代币唯一方式是核对合约地址**。这和 Handbook Citation 原则一致：每条结论必须有链上证据。

## 两个概念域的连接

```
AI 基础                        Web3 基础
────────────────────────────────────────────
LLM（预测引擎）      ←──→     Smart Contract（确定执行）
Prompt（约束空间）    ←──→     Transaction（结构化参数）
Context Window       ←──→     Block Explorer（证据追溯）
Agent（LLM+工具）    ←──→     Wallet（签名+确认）
Tool Use             ←──→     RPC / Contract Read/Write
```

两边核心原则一致：**AI 侧要防幻觉，靠 Citation 和工具验证；Web3 侧要防盲签，靠 Explorer 和链上证据。**
