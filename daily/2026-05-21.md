## AI 基础概念整理

### 1. LLM（Large Language Model）

LLM 本质是一个**基于海量文本训练的高维词汇概率系统**。它不是"会思考的大脑"，而是一个极度擅长"根据上文猜下一个词"的预测引擎。GPT、Claude、DeepSeek 这些模型，底层都是同一个原理：给定前面的词，预测下一个最可能的词。反复滚动，就生成了有逻辑的文本。

**反常识点**：
- LLM 并不"理解"你说的内容——它只计算概率分布。
- 同一个 prompt 多次运行，输出可能不同（因为采样温度 `temperature` 有随机性）。
- LLM 没有"真实世界知识"——它只知道训练数据中出现过的统计模式，所以会编造事实（幻觉）。

**实际例子**：
用 `deepseek-chat` 写 shell 脚本时，它生成过一个 `find . -delete` 命令，这种写法如果用在生产环境会直接删库。LLM 不懂"这个命令危险"，它只是觉得 `find ... -delete` 在训练数据中经常出现在一起。所以需要人审再执行。

**复核方式**：把 LLM 生成的代码/文本，再用另一条消息问"这段代码有什么安全隐患？"做交叉验证。

---

### 2. Prompt（提示词）

Prompt 不是"问问题"，而是**给 LLM 设置一个上下文约束空间**。Prompt 的质量决定了模型在"下一个词预测"这个无限可能中，被收敛到哪个方向。

一个优秀的 prompt = 角色 + 上下文 + 任务 + 格式约束 + 示例。缺了任何一块，模型就会用训练数据中的默认模式来填补——而那个默认模式大概率不是你想要的。

**常见误区**：
- ❌ 越短越好 → 足够的上下文比简洁更重要
- ❌ 一次性搞定 → prompt 经常需要迭代 3-5 轮才能稳定
- ❌ 加 "请严格遵循" 就有用 → 示例（few-shot）比指令管用得多

**对比举例**：

弱 prompt：
```
写一个 Python 函数反转字符串。
```
可能输出 `return s[::-1]` 但没有类型提示、docstring、错误处理。

强 prompt：
```
写一个 Python 函数反转字符串。要求：
1. 使用函数签名 def reverse_str(s: str) -> str
2. 包含完整的 docstring（参数，返回值，例子）
3. 处理空字符串和 Unicode 代理对
4. 不使用 s[::-1]（手动循环实现）
```
后者输出的代码直接可用。

### 3. Context Window（上下文窗口）

Context window 是 LLM **一次能"看到"的 token 总量**。它不是缓存，不是内存——每一次对话，模型都把整个上下文从头读一遍，然后生成下一个 token。

**关键认知**：
- 每一条消息 + 每一次回复，都在消耗这个窗口。
- 窗口满了，**最早的内容会被截断**（或丢弃），模型就"忘记"了对话开头的内容。
- 这就是为什么长对话末尾的模型表现会明显下降——指令可能已经在窗口外面了。

**管理技巧**：
- 关键指令放在对话**末尾**而不是开头（最新内容不会被裁）
- 长文档用参考方式而不是直接粘贴（MCP 文件读取 vs 全文粘贴）
- 定期 reset 并在新对话中植入关键 context（persistent memory 的作用就在这里）

### 4. Agent（智能代理）

Agent 不是"更聪明的 LLM"。Agent = **LLM + 工具 + 循环** 的复合体。

| | LLM | Agent |
|---|---|---|
| 做了什么 | 生成文本 | 观察 → 思考 → 行动 → 观察循环 |
| 能做什么 | 回答问题 | 执行命令、调用工具、自主决策 |
| 有记忆吗 | 只有 context window | 可以有持久化记忆 |
| 有工具吗 | 无 | 文件系统、代码执行、网络搜索等 |

**实际工作流（以 Hermes Agent 为例）**：
1. 用户说："检查 GitHub 上有没有新 issue"
2. Agent 的 LLM 分析这个请求 → 决定调用 `terminal` 工具运行 `gh issue list`
3. 看到输出 → 决定可能还需要 `search_files` 查看本地配置
4. 综合所有信息 → 生成最终回答

**常见误解**：
- ❌ Agent 就是 LLM 加了个壳 → Agent 的精髓是**自主决策循环**和**工具编排**
- ❌ Agent 能完全自主运行 → 需要清晰的边界和约束
- ❌ Agent 越"聪明"越好 → 一个有误导性的 Agent 比笨但安全的更危险

### 5. Tool Use（工具调用）

Tool use 是 **LLM 与外部系统交互的桥梁**。单独的 LLM 是"关在一个黑屋里的一台打印机"——它能生成很好的文本，但对外界一无所知。Tool use 给它开了几扇窗户。

**Hermes Agent 的工具模型**：
```
用户输入 → LLM 分析 → 选择工具 → 调用工具 → 
工具结果回到 LLM → LLM 生成最终回复
```

这个循环在模型看来是这样的：
1. 模型看到你的指令
2. 模型生成一个特殊的格式（如 JSON 格式的 function call）
3. 运行时解析这个格式，执行对应的工具
4. 结果以系统消息的形式"喂"回给 LLM
5. LLM 基于工具输出继续生成回答

**关键认知**：**LLM 本身并不能真正执行任何操作**。它只是说"我建议调用这个工具，参数是这些"。执行权在 Agent 运行时中。这意味着：
- **安全边界**：如果 prompt 注入，模型可能建议危险的命令。Agent 运行时需要自己的安全策略。
- **可靠性**：模型可能会编造工具的名字（幻觉），好的 Agent 需要函数定义验证。
- **延迟**：每次工具调用就是一个完整的 LLM 回合，多个工具调用会显著增加响应时间。

### 6. AI Coding

AI Coding 不是我让 AI 写代码那么简单。它是一种**新型人机协作范式**。

**模式分类**：

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| 补全（Completion） | 写一行提示，AI 补几行 | IDE 插件（Copilot、Codeium） |
| 对话（Chat） | 描述需求，AI 生成完整函数/文件 | 明确但重复的任务 |
| Agent 模式 | AI 自主：读代码→改代码→运行→修复 | 复杂任务、多文件改动 |
| 审查（Review） | AI 对已有代码提意见 | PR Review、安全审查 |

**误区 1：AI 写的代码可以直接用** → 事实：AI 代码正确率大概在 70-80%。剩余的包括：API 版本不对、类型错误、忘了 import、逻辑边界有 bug、安全隐患。必须修改和测试。

**误区 2：AI 适合一切语言和框架** → 事实：AI 在主流语言和框架（Python、React、TypeScript、Django）上表现最好。小众或冷门框架的表现会断崖式下降。

**误区 3：prompt 越详细越好** → 事实：太长且结构混乱的 prompt 反而会稀释关键信息。结构化 > 详细化。用 bullet point 分条，每条不超过一句话。

**最佳实践 prompt 模板**：
```
# 角色
你是一个资深 Python 后端工程师

# 任务
重构这个函数，让它使用 async/await

# 约束
- 保持相同的函数签名
- 保留所有 docstring
- 只修改实现逻辑

# 上下文
文件：/path/to/handler.py
```
