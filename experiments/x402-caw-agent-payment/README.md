# x402 Paywall + Cobo CAW Agent Payment Demo

This is a minimal runnable demo for Week 2 Module B. It shows an agent calling an x402-protected AI API, receiving a `402 Payment Required` response, checking a CAW/Pact-style policy, creating a payment payload, retrying the request, settling the payment, and receiving the protected result.

The demo is local and dependency-free. It uses Node.js built-in modules only. The CAW signer, Pact engine, x402 facilitator, and settlement ledger are mocked so the full flow can run without real funds, private keys, or external API credentials.

## What It Demonstrates

- Service provider exposes `POST /v1/infer`.
- Provider returns `402 Payment Required` when no payment is attached.
- Agent parses the x402-style payment requirement.
- Pact policy checks:
  - allowed chain
  - allowed asset
  - allowed API resource
  - allowed recipient
  - per-payment cap
  - daily budget
  - time window
  - payment count
- CAW mock signer creates a signed payment payload.
- Provider verifies and settles the payment into a local ledger.
- Agent receives the final protected API result.
- Every important step is written to `audit-log.jsonl`.

## Run

From this directory:

```bash
node src/demo.js
```

Or from the repo root:

```bash
node experiments/x402-caw-agent-payment/src/demo.js
```

Expected output:

```text
x402 provider listening on http://127.0.0.1:4020/v1/infer

Agent received final API response:
```

## Files

| File | Purpose |
| --- | --- |
| `src/demo.js` | Provider, agent, Pact checks, CAW mock signer, facilitator, settlement |
| `pact-policy.json` | Budget, recipient, API, chain, asset, and time-window policy |
| `audit-log.jsonl` | Generated audit trail after running the demo |
| `settlement-ledger.json` | Generated settlement receipt ledger after running the demo |

## Security Boundary

This demo intentionally does not perform real onchain payment. In a production implementation:

- The payment payload should be generated by CAW / MPC-controlled signing.
- Pact should be enforced by the wallet/signing layer, not only by the agent.
- x402 verification and settlement should use the official facilitator flow.
- Secrets, private keys, API keys, and session keys must never enter the audit log.
- Budget increases, new recipients, unknown contracts, unlimited approvals, and policy changes must require human approval.
