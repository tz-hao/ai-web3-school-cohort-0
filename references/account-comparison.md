# EOA vs 智能账户 vs 多签账户：权限边界对比

> 理解「谁能发起交易、谁能批准、谁承担风险」是 Agent Wallet 和 Agent Identity 章节的前置基础。

---

## 1. EOA（Externally Owned Account，外部账户）

### 一句话

一把私钥 = 一个地址 = 全部控制权。最简单也最危险。

### 谁持有控制权

**私钥持有者**。谁有私钥，谁就是这个地址的绝对主人。没有恢复机制，没有审批流程，没有限额。

### 谁可以发起交易

私钥持有者可以发起任意交易——转 ETH、调合约、approve 无限额度、撤销授权。**不需要任何人同意。**

### 是否支持多人确认

**不支持。** 一把钥匙开一扇门。你无法让两个人同时控制一个 EOA（除非共享私钥，那是灾难）。

### 是否支持恢复、限额或自动化策略

**完全不支持。**
- 私钥丢了 → 资产永久锁定。没有「忘记密码」，没有客服。
- 想设每天最多转 1 ETH？做不到。EOA 不知道「限额」是什么。
- 想让 Agent 自动执行定投？你需要把私钥交给 Agent——Agent 被攻破 = 全部资产归零。

### 典型使用场景

- 个人日常钱包（MetaMask 默认账户）
- 短期交互（领空投、mint NFT）
- 开发者测试（Hardhat 默认账户、Anvil 账户）

### 主要风险点

| 风险 | 说明 |
|------|------|
| 单点失败 | 私钥泄露 = 资产全丢，不可逆 |
| 无恢复 | 助记词丢了就永远丢了 |
| 无审计 | 事后无法追溯是谁签的名（除非主动保存记录） |
| Agent 不友好 | 给 Agent 私钥 = Agent 可以掏空你。没有权限分级 |

### 安全边界

> EOA 的边界是一条线：**线的这边是你，那边是全世界。线断了，一切归零。**

## 2. Smart Account（智能账户 / ERC-4337）

### 一句话

账户本身是一段可编程代码。你可以自定义「谁可以做什么、花多少钱、在什么条件下」。

### 谁持有控制权

**Owner（可以是 EOA，也可以是另一个合约）+ 账户代码中的 Policy**。Owner 设定规则，Policy 执行规则。Owner 不需要每笔交易都签名。

### 谁可以发起交易

三层：
1. **Owner** — 可以改规则、恢复账户、撤销权限。最高权限。
2. **Session Key** — 临时授权，在限定时间和额度内可以发起交易，不需要每次签名。
3. **自动策略** — 合约代码中预设的自动化规则（定时、定频、定条件），无需人工触发。

### 是否支持多人确认

**可以编程实现。** 比如设置规则：转账 > 1 ETH 需要 Owner 二次确认，≤ 1 ETH 自动放行。但这和原生多签（每个 signer 都是独立 EOA）不同——智能账户的多人确认是代码层面的策略，不是独立身份。

### 是否支持恢复、限额或自动化策略

**全面支持：**
- **恢复**：可以设置多个 Guardian（守护者），多数 Guardian 同意即可恢复被攻破的账户
- **限额**：每日、单笔、白名单合约均可设额度
- **自动化**：定时任务、条件触发（价格跌到 X 就补仓）、AI Agent 定期调仓

### 典型使用场景

- AI Agent 钱包（Agent 拿 Session Key，不能动大额）
- 团队运营钱包（有额度，有审计日志）
- 高频交易机器人（Session Key 免去每笔签名）
- 新手账户（有 Guardian 可恢复、有风险合约黑名单）

### 主要风险点

| 风险 | 说明 |
|------|------|
| 合约 bug | 你的账户规则是一段代码——代码有 bug，规则就废了 |
| Owner 密钥仍可丢 | Owner 通常还是 EOA，丢失意味着无法恢复或改规则 |
| Policy 配置错误 | 限额设太高 = 有没有 Session Key 差别不大 |
| 链上复杂度 | 牵扯 EntryPoint、Paymaster、Bundler，比 EOA 多很多依赖 |

### 安全边界

> 智能账户的边界是一圈栅栏：**你可以控制栅栏的高度、门的宽度、谁有钥匙、什么时间开门。但栅栏本身是代码——代码有 bug，栅栏就有洞。**

## 3. Multi-sig Account（多签账户，以 Safe 为代表）

### 一句话

N 个人中需要 M 个人签名，交易才能执行。不是一个人说了算。

### 谁持有控制权

**多个 Owner（各自持有独立的 EOA 私钥），按 M-of-N 阈值集体控制。** 没有一个 Owner 可以单独行动。

### 谁可以发起交易

任何 Owner 可以发起（提交）交易。但交易不会立刻执行——它进入待确认队列，等待足够多的 Owner 签名。只有达到阈值（比如 3/5），交易才真正上链。

### 是否支持多人确认

**核心功能。** 常见配置：
- 2/3：两人公司，两个 Founder
- 3/5：DAO 金库
- 1/2：个人+配偶的应急账户
- 阈值可调——但这本身是一笔链上交易（需要当前 Owner 确认）

### 是否支持恢复、限额或自动化策略

**部分支持（取决于具体实现）：**
- **恢复**：如果一个 Owner 丢了私钥，其他 Owner 可以提案移除旧 Owner、添加新 Owner
- **限额**：Safe 支持设置单笔最大额度，但不像智能账户那样灵活可编程
- **自动化**：不原生支持 Session Key 或自动化策略。多签的每次操作都需要 N 个人手动确认——这是安全代价

### 典型使用场景

- DAO 金库（社区共同管理协议收入）
- 团队运营资金（防止单人卷款跑路）
- 协议管理员密钥（升级合约需要 3/5 核心开发者签名）
- 高风险权限账户（一次操作可能影响千万资产）

### 主要风险点

| 风险 | 说明 |
|------|------|
| 签名者不和 | 阈值达不到，资金锁死 |
| 签名者串通 | M 个恶意 signer 可以掏空，剩下的 N-M 个无能为力 |
| 签名疲劳 | 每次操作都要集齐人 → 人会偷懒，懒了就盲签 |
| 延迟风险 | 紧急情况（预言机暴跌、需要紧急撤资）时凑不够签名 |
| 不适合自动化 | Agent 无法单独操作多签，每次都要等人 |

### 安全边界

> 多签的边界是一排锁：**必须 M 把钥匙同时转动，门才开。好处是单点不崩，代价是敏捷性为零。**

## 三种账户的对比总结

| 维度 | EOA | 智能账户 | 多签账户 |
|------|-----|---------|---------|
| 控制权 | 1 私钥 = 全额控制 | Owner + Policy 代码 | M/N 阈值集体控制 |
| 发起交易 | 私钥持有者 | Owner / Session Key / 自动策略 | 任何 Owner 提交，等够签名后执行 |
| 多人确认 | ❌ | 可编程实现 | ✅ 原生支持 |
| 恢复机制 | ❌ 丢则永久 | ✅ Guardian 恢复 | ✅ 其他 Owner 可替换 |
| 限额策略 | ❌ | ✅ 灵活可编程 | ⚠️ 有限支持 |
| 自动化友好 | ❌ 全或无 | ✅ Session Key + 策略 | ❌ 每次要等人 |
| 适合 Agent | ❌ 危险 | ✅ 理想选择 | ❌ 敏捷性为零 |
| 适合个人 | 日常小额 | 长期主力 | 一般不需要 |
| 适合团队 | ❌ | ⚠️ 需要信任 Owner | ✅ 标准方案 |
| 适合 DAO | ❌ | ⚠️ 不如多签 | ✅ 首选 |
| 主要风险 | 单点失败 | 合约 bug | 签名协调失败 |

## AI × Web3 Bridge 视角：为什么 Agent 需要智能账户

Agent 的三个核心需求，只有智能账户能同时满足：

```
Agent 需求           EOA    智能账户    多签
─────────────────────────────────────────────
自动执行交易          ✅      ✅          ❌
权限分级（只花小钱）   ❌      ✅          ❌
需要时人工接管         ❌      ✅          ✅
可恢复 / 可撤销       ❌      ✅          ✅
```

**结论**：把私钥交给 Agent = 把全部资产控制权交给一段代码。**智能账户的 Session Key 机制让 Agent 在有限时空内花有限的钱**——这是 Agent Wallet 章节的核心前提。
